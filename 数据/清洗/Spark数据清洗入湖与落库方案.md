# Spark 数据清洗入湖与落库方案

## 1. 文档目的

本文档用于说明三源招聘岗位数据在爬取完成后的清洗、标准化、HDFS 入湖和数据库落库方案。当前数据来源包括前程无忧、智联招聘、国家大学生就业服务平台，已通过前置融合脚本生成统一总表：

```text
D:\桌面\毕业设计\数据\爬虫\总爬虫数据分析\分析结果\总爬取数据.csv
```

当前统一总表概况如下：

| 指标 | 数值 |
| --- | --- |
| 原始 CSV 文件数 | 240 |
| 去重前有效岗位行数 | 5993 |
| 移除重复岗位行数 | 136 |
| 去重后统一总表行数 | 5857 |
| 全局技能覆盖率 | 90.8% |
| 全局薪资解析覆盖率 | 95.9% |
| 全局岗位描述覆盖率 | 100.0% |

后续清洗目标是将统一总表进一步处理为适合分析、知识图谱构建、可视化展示和数据库查询的标准数据集。

## 2. 总体清洗思路

整体采用 Bronze、Silver、Gold 三层数据分层结构：

| 层级 | 作用 | 主要内容 | 存储位置 |
| --- | --- | --- | --- |
| Bronze 原始层 | 保留原始数据与追溯信息 | 三源原始 CSV、统一融合 CSV、来源文件、行号、指纹 | HDFS |
| Silver 清洗层 | 形成标准化岗位宽表 | 字段类型转换、空值处理、岗位/城市/薪资/学历/经验/日期标准化 | HDFS Parquet |
| Gold 应用层 | 面向分析和展示 | 岗位技能关系、技能频次、城市岗位薪资、岗位需求统计 | HDFS Parquet + 数据库 |

清洗原则：

1. Bronze 层尽量不删除字段，用于数据追溯。
2. Silver 层保留分析所需字段，并保留部分质量与来源字段。
3. Gold 层只保留可视化、统计分析和图谱构建需要的字段。
4. 数据库主表不存放大量长文本，长文本单独进入详情表。
5. 数据库不承担原始数据留存职责，原始数据和完整清洗数据以 HDFS 为准。

## 3. HDFS 目录设计

建议 HDFS 根目录使用通俗中文名称：

```text
/数据/就业技能需求分析/
```

目录结构如下：

```text
/数据/就业技能需求分析/
  01_原始数据层/
    融合原始岗位表/批次日期=2026-06-29/

  02_清洗标准层/
    岗位清洗宽表/批次日期=2026-06-29/城市=北京/
    岗位文本详情表/批次日期=2026-06-29/

  03_分析应用层/
    岗位技能关系表/批次日期=2026-06-29/岗位=数据分析师/
    技能热度统计表/批次日期=2026-06-29/岗位=数据分析师/
    岗位专业关系表/批次日期=2026-06-29/专业=计算机类/
    岗位需求汇总表/批次日期=2026-06-29/
    城市岗位薪资表/批次日期=2026-06-29/城市=北京/
    学历经验统计表/批次日期=2026-06-29/
```

其中，`01_原始数据层` 对应 Bronze 层，`02_清洗标准层` 对应 Silver 层，`03_分析应用层` 对应 Gold 层。目录名前加数字是为了在 HDFS 或文件管理工具中按处理顺序排序。

分区建议：

| 中文目录名 | 对应英文表名 | 分区字段 | 说明 |
| --- | --- | --- | --- |
| `融合原始岗位表` | `unified_jobs_raw` | `批次日期` | 按采集批次保存原始融合结果 |
| `岗位清洗宽表` | `job_posting_clean` | `批次日期`, `城市` | 常按城市分析，适合城市分区 |
| `岗位文本详情表` | `job_text_detail` | `批次日期` | 保存岗位描述、任职要求、职责、技能证据等长文本 |
| `岗位技能关系表` | `job_skill_relation` | `批次日期`, `岗位` | 常按岗位分析技能，适合岗位分区 |
| `技能热度统计表` | `skill_frequency` | `批次日期`, `岗位` | 保存不同岗位、城市下的技能出现频次 |
| `岗位专业关系表` | `job_major_relation` | `批次日期`, `专业` | 保存岗位与专业要求之间的关系，便于按专业汇总 |
| `城市岗位薪资表` | `city_job_salary` | `批次日期`, `城市` | 保存城市和岗位维度的薪资统计 |
| `岗位需求汇总表` | `job_demand_summary` | `批次日期` | 保存城市、岗位、来源维度的岗位数量 |
| `学历经验统计表` | `education_experience_summary` | `批次日期` | 保存学历和经验要求统计 |

## 4. Spark 清洗流程

### 4.1 输入数据

优先使用统一融合总表作为清洗入口：

```text
D:\桌面\毕业设计\数据\爬虫\总爬虫数据分析\分析结果\总爬取数据.csv
```

如需保留更完整的原始追溯，也可以同时将三源原始 `*_jobs.csv` 上传到 `01_原始数据层`（Bronze 层）。

### 4.2 清洗步骤

1. 使用 Spark 读取统一 CSV。
2. 去除全空行和明显无效行。
3. 统一字段命名，将中文字段映射为数据库友好的英文字段。
4. 转换字段类型，包括数值、日期、时间戳等字段；经纬度因缺失较多，不进入清洗主表。
5. 标准化岗位名称、城市、学历、经验、岗位类型、公司规模、公司性质。
6. 处理薪资字段，生成月薪下限、月薪上限、月薪中位数和估算年薪。
7. 对技能字段进行数组化处理，并拆分生成岗位-技能关系表。
8. 按去重键或内容指纹进行二次去重。
9. 输出 Silver 层岗位宽表和岗位详情表。
10. 基于 Silver 层生成 Gold 层统计表。
11. 将 Gold 层和部分 Silver 层结果通过 JDBC 写入数据库。

### 4.3 去重规则

优先级如下：

1. 优先使用 `去重键`。
2. 若 `去重键` 缺失，使用 `内容指纹`。
3. 若指纹字段均缺失，则使用以下字段拼接后生成 SHA256：

```text
岗位名称 + 公司名称 + 标准城市 + 岗位描述清洗文本 + 来源链接
```

重复记录保留规则：

1. 优先保留 `完整度评分` 更高的记录。
2. 完整度评分相同时，优先保留来源权重较高的数据。
3. 来源权重建议为：

| 来源 | 权重 |
| --- | --- |
| 国家大学生就业服务平台 | 3 |
| 前程无忧 | 2 |
| 智联招聘 | 1 |

该权重仅用于重复记录保留，不代表数据质量绝对高低。

### 4.4 是否需要分组清洗

本项目不建议将数据按专业或岗位拆成多个独立脚本分别清洗。推荐做法是：清洗阶段统一处理全量岗位数据，输出阶段再按岗位、城市或专业进行分区和聚合。

原因如下：

1. Spark 擅长对全量数据做并行处理，手工拆成多个专业分别清洗会增加脚本数量和维护成本。
2. 薪资、学历、经验、日期、空值、去重等清洗规则应保持统一，否则不同专业之间的统计口径会不一致。
3. 当前数据量约 5857 条，远未达到必须拆批清洗的规模，全量一次清洗更简单可靠。
4. 后续分析可以通过 `标准岗位`、`正文确认标准专业`、`标准城市` 进行分区、分组和聚合，不需要在清洗入口阶段物理拆分。

建议采用以下处理方式：

| 阶段 | 是否分组 | 建议做法 |
| --- | --- | --- |
| 数据清洗 | 不按专业拆脚本 | 全量读取、统一规则、统一去重、统一字段标准化 |
| HDFS 存储 | 可以分区 | `岗位清洗宽表` 可按 `批次日期/城市` 分区，技能关系表可按 `批次日期/岗位` 分区 |
| 专业分析 | 单独拆表 | 将 `正文确认标准专业` 拆成 `岗位专业关系表`，用于专业要求统计 |
| 岗位分析 | 分组统计 | 使用 `groupBy(标准岗位, 标准城市)` 生成岗位需求、薪资和技能统计 |

如果这里的“专业”指的是招聘岗位方向，例如数据分析、算法、大数据开发，则应使用 `标准岗位` 作为主要分组字段。如果指的是学历专业要求，例如计算机类、软件工程、数据科学，则应从 `正文确认标准专业` 拆出单独的专业关系表。

### 5.4 薪资字段

| 原字段 | 清洗方式 | Silver 层 | 数据库主表 |
| --- | --- | --- | --- |
| `薪资原文` | 保留原始文本，进入详情表 ，空白填充 `面议`| 保留 | 详情表 |
| `最低月薪` | 转 double，单位统一为元/月 | 保留 | 保留 |
| `最高月薪` | 转 double，单位统一为元/月 | 保留 | 保留 |
| `薪资中位数` | 转 double，使用 `(最低月薪 + 最高月薪) / 2` | 保留 | 保留 |
| `年薪月数` | 转 int，空值可填 12，也可保留 null | 保留 | 保留 |
| `估算年薪` | 新增派生字段，使用 `薪资中位数 × 年薪月数`；年薪月数缺失时按 12 个月估算 | 保留 | 保留 |

薪资异常处理规则：

1. `最低月薪` 或 `最高月薪` 小于 1000 元时置空。
2. `最低月薪` 或 `最高月薪` 大于 200000 元时置空。
3. 若 `最低月薪 > 最高月薪`，交换两者。
4. 若只有一个薪资值，则 `薪资中位数` 使用该值。
5. 若薪资为面议，则数值字段置空，`薪资原文` 保留。
6. 新增 `估算年薪` 字段，计算公式为：`薪资中位数 × coalesce(年薪月数, 12)`。
7. 如果 `薪资中位数` 为空，则 `估算年薪` 置空，不使用行业平均值强行填补。
8. `估算年薪` 主要用于薪资分布、城市薪资对比和岗位薪资对比，论文中应说明其为基于招聘薪资字段的估算值。

### 5.7 岗位类型与日期字段

| 原字段 | 清洗方式 | Silver 层 | 数据库主表 |
| --- | --- | --- | --- |
| `岗位类型` | 统一为 `全职`、`实习`、`校招`、`兼职`、`未知` | 保留 | 保留 |
| `发布日期` | 转 date，异常值置空 | 保留 | 保留 |
| `发布日期原文` | 原始文本，仅追溯 | 保留 | 删除 |
| `采集时间` | 转 timestamp，并生成批次日期 | 保留 | 保留 |

日期处理规则：

1. `发布日期` 格式统一为 `yyyy-MM-dd`。
2. `采集时间` 格式统一为 Spark timestamp。
3. 如果发布日期晚于采集时间，标记为异常，日期字段可置空。
4. 如果发布日期缺失，后续按采集批次进行统计，不强行补发布日期。

### 5.8 专业字段

| 原字段 | 清洗方式 | Silver 层 | 数据库主表 |
| --- | --- | --- | --- |
| `正文确认标准专业` | 按 `|` 拆分，可形成岗位-专业关系表 | 保留 | 可选 |
| `专业类别` | 可拆为专业分类维表 | 保留 | 删除 |
| `专业要求级别` | 统一为 `要求`、`优先`、`不限`、`未知` | 保留 | 可选 |

专业字段在当前系统中不是最核心的分析对象，建议 Silver 层保留，Gold 层可按需要生成专业要求统计表。

如果需要“同一个专业放一起”，不建议在原始 CSV 阶段手工拆文件，而建议在 Spark 清洗后生成 `岗位专业关系表`：

1. 将 `正文确认标准专业` 按 `|` 拆分为专业数组。
2. 使用 `explode` 将一条岗位记录拆成多条岗位-专业关系。
3. 输出到 `03_分析应用层/岗位专业关系表/批次日期=.../专业=.../`。
4. 后续按专业统计岗位数量、技能要求、学历要求和薪资水平。

这样既可以实现“同一个专业的数据放在同一个分区”，也能避免在清洗入口阶段拆分数据导致规则不一致。

### 5.10 技能字段

## 8. 数据库落库方案

数据库建议使用 MySQL 或 PostgreSQL。HDFS 保存完整数据，数据库只保存系统展示和查询需要的数据。

建议落库表：

| 表名 | 来源层级 | 作用 |
| --- | --- | --- |
| `job_posting_clean` | Silver | 岗位主查询表 |
| `job_posting_detail` | Silver | 岗位详情与长文本 |
| `job_skill_relation` | Gold | 知识图谱和技能分析 |
| `skill_frequency` | Gold | 技能热度分析 |
| `job_major_relation` | Gold | 专业要求分析 |
| `city_job_salary` | Gold | 薪资分析 |
| `education_experience_summary` | Gold | 学历经验统计 |
| `job_demand_summary` | Gold | 城市岗位需求统计 |

落库方式：

1. Spark 清洗后先写 HDFS Parquet。
2. 再从 Parquet 读取 Gold 表。
3. 使用 Spark JDBC 写入数据库。
4. 写入模式建议开发阶段使用 `overwrite`，稳定后使用 `append` 或按 `batch_date` 分区覆盖。

JDBC 写入示例：

```python
df.write \
  .format("jdbc") \
  .option("url", "jdbc:mysql://localhost:3306/job_skill") \
  .option("dbtable", "job_posting_clean") \
  .option("user", "root") \
  .option("password", "password") \
  .mode("overwrite") \
  .save()
```

## 11. 当前数据风险与处理建议

1. 国家大学生就业服务平台的经验字段缺失率较高，不建议强行补为 `不限`，应标记为 `未知` 或保留 null。
2. 当前数据主要集中在 2026 年 6 月附近，更适合做横截面分析；趋势预测需要历史数据或后续多批次采集。
3. `算法工程师` 样本量明显高于其他岗位，做总体统计时应注意岗位样本不均衡问题。
4. 技能字段已有较高覆盖率，但仍有约 9.2% 缺失，可在后续 NLP 抽取阶段补充。
5. 商业招聘平台字段更完整，但合规与来源解释需要在论文中说明仅用于毕业设计研究。
